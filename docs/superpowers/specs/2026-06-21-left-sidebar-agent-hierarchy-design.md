# 左侧栏 agent 分层与聊天标题管理设计

日期：2026-06-21

## 1. 背景

当前前端左侧栏基于 `frontend/src/components/layout/LeftSidebar.vue` 中的 `Conversations` 分组列表渲染，会话可以按 agent 归组，但整体层级关系仍偏扁平，不能很好地表达“聊天标题属于某个 agent”的结构。当前实现也缺少以下能力：

- 聊天标题作为 agent 的二级层级展示不够明显
- 每个 agent 下默认仅展示少量会话并逐步展开
- 搜索聊天标题时仍保持 agent 分组关系
- 对聊天标题进行置顶，并将置顶状态持久化到后端

本次需求目标是将左侧栏重构为更清晰的两级导航结构，同时补充搜索、置顶和分组内逐步展开能力。

## 2. 目标

实现以下用户可见行为：

1. 左侧栏以 agent 为一级节点、聊天标题为二级节点展示，层级关系明显，聊天标题相对于 agent 有可感知缩进。
2. 每个 agent 默认最多显示 5 个聊天标题。
3. 点击“显示更多”后，该 agent 再额外显示 20 个聊天标题；该展开状态只对当前页面有效，刷新后恢复默认值。
4. 支持搜索所有聊天标题，搜索后仍按 agent 分组展示结果，不打平成单列表。
5. 支持对聊天标题置顶，置顶后的会话仍保留在原 agent 下，但排序到该 agent 内最前面。
6. 置顶状态存储在后端，而不是只存在前端本地。

## 3. 不做范围

本次设计明确不包含以下内容：

- 不增加全局“置顶专区”
- 不支持跨 agent 拖拽排序
- 不支持会话手动自定义排序
- 不为“显示更多”状态做持久化
- 不引入数据库迁移框架或兼容旧版本迁移逻辑

## 4. 方案选择

### 4.1 备选方案

- 方案 A：继续基于 `Conversations` 组件增强
- 方案 B：保留现有 store 和会话数据流，但左侧列表改为自行渲染
- 方案 C：先拆出多个 sidebar 子组件后再大规模重做

### 4.2 最终采用方案

采用方案 B。

原因：

- 需求核心已从简单分组展示转向“明确层级 + 组内控制 + 搜索 + 置顶”的专用导航结构。
- 现有 `Conversations` 更适合通用分组会话列表，不适合自然表达“每组默认 5 条 + 点击后追加 20 条 + 搜索后仍保留分组 + 组内置顶排序”。
- 自定义渲染可以更稳定地控制层级关系、缩进、按钮位置、空状态和后续扩展。

## 5. 前端信息架构

左侧栏从上到下分为四块：

1. 顶部 header
2. 搜索区
3. agent 分组区
4. 每个 agent 下的聊天标题子列表

### 5.1 顶部 header

保留现有能力：

- 标题 `Chats`
- 全部分组展开/折叠
- 侧边栏收起/展开

### 5.2 搜索区

在 header 下方新增搜索输入框。

行为规则：

- 输入时实时过滤聊天标题
- 搜索范围是所有聊天标题，而不是当前已显示的 5 条
- 搜索只匹配会话标题，不匹配 agent 名称
- 搜索结果仍按 agent 分组展示
- 没有命中任何标题的 agent 分组不显示

### 5.3 agent 分组区

每个 agent 为一个一级节点，展示：

- 展开/折叠箭头
- agent 名称
- 数量信息

数量信息规则：

- 非搜索状态下，显示该 agent 总会话数
- 搜索状态下，显示命中数；如需要可采用“命中数/总数”格式，但实现阶段以命中数优先

### 5.4 聊天标题子列表

聊天标题为 agent 的二级节点，视觉上相对 agent 头部有明确缩进。

每个标题项支持：

- 切换当前会话
- 重命名
- 删除
- 置顶/取消置顶

## 6. 视觉层级设计

为了强化“聊天标题属于 agent”的关系，左侧栏视觉分层按以下原则实现：

### 6.1 agent 头部

- 采用明显的块级区域样式
- 使用更强的背景对比或卡片感
- 名称使用更高字重
- 折叠箭头固定在左侧
- 数量 badge 较小，作为辅助信息

### 6.2 聊天标题项

- 整体相对 agent 头部右缩进
- 可配合淡色引导线、圆点或间距增强层级感
- 标题字号略小于 agent 名称
- hover 与 active 状态仅作用于标题项自身

最终目标是使结构接近“一级目录：agent，二级目录：chat title”，而非普通扁平 grouped list。

## 7. 排序与过滤规则

### 7.1 分组规则

所有会话先按 `agent_id` 分组，并使用前端已有 agent store 提供的 agent 名称进行显示。

### 7.2 组内排序规则

每个 agent 分组内，会话按以下顺序排序：

1. `is_pinned = true` 的会话在前
2. 在已置顶会话中，按 `pinned_at` 倒序
3. 其余会话按 `updated_at` 倒序

这样可以保证：

- 置顶项稳定地位于组内前部
- 新近置顶项排在更前面
- 普通会话仍保持最近更新优先

### 7.3 搜索过滤规则

- 对全部会话标题做不区分大小写的包含匹配
- 先完成过滤，再将命中结果按 agent 分组
- 每个 agent 只显示命中项
- 搜索结果仍应用同样的组内排序规则

## 8. “显示更多”规则

每个 agent 单独维护当前可见数量。

初始规则：

- 初始可见数 = 5

点击规则：

- 每点击一次“显示更多”，当前 agent 的可见数 +20
- 若剩余项不足 20，则直接显示到全部

重置规则：

- 刷新页面后重置为每组 5 条
- 搜索关键词变化时，也重置各 agent 当前显示数量为 5

显示条件：

- 某个 agent 当前匹配结果数大于可见数时，才显示“显示更多”按钮
- 已全部显示时，不再显示按钮

## 9. 前端状态设计

### 9.1 保留现有 store

保留 `frontend/src/stores/sessions.ts` 作为会话数据来源，不新增独立 sidebar store。

### 9.2 LeftSidebar 组件内部状态

建议在 `LeftSidebar.vue` 中维护以下 UI 状态：

- `searchQuery`：搜索关键词
- `collapsedGroups`：已折叠的 agent 名称集合
- `visibleCountByAgent`：每个 agent 当前可见条数

### 9.3 衍生数据

由 `sessionsStore.sessions` 计算得到 sidebar 渲染模型：

1. 读取全部 sessions
2. 基于 `searchQuery` 过滤标题
3. 按 agent 分组
4. 在组内应用 pinned + 时间排序
5. 结合 `visibleCountByAgent` 生成每组当前实际展示项

## 10. 后端数据模型设计

### 10.1 Session 表新增字段

在 `lc_agent/db/models.py` 的 `SessionMeta` 中新增：

- `is_pinned: bool = False`
- `pinned_at: datetime | None = None`

### 10.2 字段意义

- `is_pinned`：表示当前会话是否处于置顶状态
- `pinned_at`：记录最后一次被置顶的时间，用于已置顶会话之间的稳定排序

### 10.3 数据更新规则

- 当用户执行置顶时：
  - `is_pinned = true`
  - `pinned_at = 当前时间`
- 当用户取消置顶时：
  - `is_pinned = false`
  - `pinned_at = null`

## 11. API 设计

### 11.1 GET `/api/sessions`

返回的 session 数据新增：

- `is_pinned`
- `pinned_at`

### 11.2 PUT `/api/sessions/{session_id}`

扩展现有更新接口，请求体支持：

- `title`
- `model`
- `is_pinned`

行为规则：

- 若只更新标题或模型，则保持当前置顶状态不变
- 若更新 `is_pinned=true`，后端同时刷新 `pinned_at`
- 若更新 `is_pinned=false`，后端同时清空 `pinned_at`

本次不额外新增专门的 pin/unpin 独立接口，复用已有 session 更新接口。

## 12. 前端接口与类型变更

### 12.1 `frontend/src/api/http.ts`

扩展 `updateSession` 的入参，支持：

- `is_pinned?: boolean`

### 12.2 `frontend/src/stores/sessions.ts`

扩展 `Session` 接口字段：

- `is_pinned: boolean`
- `pinned_at: string | null`

同时新增或扩展 store 方法，用于：

- 更新单个 session 的置顶状态
- 在接口成功后同步本地 session 数据

## 13. 组件实现设计

### 13.1 `LeftSidebar.vue`

该组件将从“依赖 `Conversations` 完成主要列表渲染”调整为“组件自身负责分组与标题列表渲染”。

建议保留现有职责：

- 响应 `newChat`
- 响应 `switchSession`
- 响应 `toggleCollapse`
- 触发重命名与删除

新增职责：

- 渲染搜索输入框
- 渲染 agent 分组头
- 渲染组内标题项
- 渲染“显示更多”按钮
- 触发置顶/取消置顶

### 13.2 会话项菜单

延续当前“重命名/删除”交互模型，在同一操作菜单中新增：

- 若未置顶：`置顶`
- 若已置顶：`取消置顶`

这样能避免在标题项表面塞入过多图标，保持左侧栏紧凑与层级清晰。

## 15. 数据库与开发环境策略

项目当前处于非常早期阶段，不引入兼容旧结构的迁移包袱。

因此本次按以下原则处理：

- 直接扩展 `sessions` 表对应模型
- 不补历史兼容逻辑
- 不引入迁移框架

如果开发者本地已有旧表结构，允许通过重建本地开发数据库来对齐 schema，而不是为旧 schema 编写兼容代码。

## 16. 风险与控制

### 16.1 风险

1. 从通用 `Conversations` 列表切换到手写渲染后，需要自行保证当前重命名、删除、切换会话交互不退化。
2. 增加后端置顶字段后，前后端 session 类型必须同步，否则容易出现字段缺失或排序异常。
3. 搜索、折叠、显示更多三种状态叠加时，若状态管理不清晰，容易出现 UI 不一致。

### 16.2 控制措施

1. 复用现有 `sessionsStore` 数据流，不重做会话管理逻辑。
2. 将搜索、分组、排序、可见数量都收敛为明确的 computed 数据流。
3. 用路由测试覆盖置顶字段读写。
4. 实现完成后做人工交互验证，重点检查搜索与显示更多的组合场景。

## 17. 实施结果定义

当以下条件都满足时，视为本次需求完成：

- 左侧栏不再呈现为扁平 grouped list，而是具有明显 agent -> chat title 的二级层级关系
- 每个 agent 默认仅展示 5 条聊天标题，并支持每次点击“显示更多”后新增 20 条
- 搜索能在全量聊天标题中匹配，并保留 agent 分组展示
- 置顶状态写入后端并在刷新后仍有效
- 置顶会话保留在原 agent 下，并排到该 agent 组内最前面
